# 결정 사항 (Decisions to Lock)

**목적**: 고객 검토 미팅에서 반드시 결정을 받아야 하는 항목
**우선순위**: 다음 단계 (실제 구현) 진행을 위해 필수
**결정 방법**: 각 항목의 선택지 중 하나를 선택

---

## 결정 항목 개요

| 번호 | 항목 | 영향 범위 | 우선순위 |
|---|---|---|---|
| D1 | 9개 담보 표시 범위 | UI 구조, 데이터 로딩 | 높음 |
| D2 | 보험료 비교 기능 제공 여부 | 기능 범위, 법적 책임 | 높음 |
| D3 | 원문 보기 (PDF) 구현 우선순위 | 개발 일정, 비용 | 중간 |
| D4 | Notes 상세 기본 상태 | UX, 정보 노출 정도 | 중간 |
| D5 | 회사명 + 상품명 표시 위치 | 정보 구조 | 낮음 |
| D6 | 모바일 우선 개발 여부 | 개발 순서, 일정 | 높음 |

---

## D1: 9개 담보 표시 범위

### 왜 결정이 필요한가?

현재 예제 3에서 9개 담보를 한 번에 모두 표시하는데,
고객 피드백에서 "너무 길다"는 의견이 나올 경우 UI 구조 변경이 필요합니다.
이 결정에 따라 테이블 렌더링 로직과 데이터 로딩 전략이 달라집니다.

### 선택지

#### 옵션 1: 9개 모두 표시 (현재 방식) ⭐ 추천

**장점**:
- 한 번에 모든 정보 확인 가능
- 인쇄 / 공유 시 유리
- 추가 클릭 불필요

**단점**:
- 스크롤 많이 필요
- 모바일에서 답답함

**구현 영향**:
- 변경 없음
- 개발 일정 영향 없음

#### 옵션 2: 핵심 3~5개만 표시 + "더 보기" 버튼

**장점**:
- 첫 화면 간결
- 빠른 로딩

**단점**:
- 추가 클릭 필요
- "어떤 담보가 핵심인지" 기준 모호

**구현 영향**:
- 담보 우선순위 로직 추가 필요
- 개발 1주 추가

#### 옵션 3: 아코디언 방식 (담보별 접기/펼치기)

**장점**:
- 유연한 정보 탐색
- 관심 담보만 확인 가능

**단점**:
- 전체 비교 어려움
- 인쇄 시 불리

**구현 영향**:
- UI 컴포넌트 재설계 필요
- 개발 2주 추가

### 디폴트 추천안

✅ **옵션 1 (9개 모두 표시)**

**이유**:
- 고객 예제 요구사항에 명시된 "종합 비교"와 일치
- 프로토타입에서 부정적 피드백 없음
- 구현 복잡도 최소

## D2: 보험료 비교 기능 제공 여부

### 왜 결정이 필요한가?

보험료는 개인별 조건에 따라 달라지므로 "확정값"처럼 보이면 법적 리스크가 있습니다.
현재는 "참고용 예시"로만 표시하지만, 이 기능 자체를 제공할지 결정이 필요합니다.

### 선택지

#### 옵션 1: 제공 (현재 방식 - 참고용만) ⭐ 추천

**장점**:
- 고객이 대략적인 보험료 수준 파악 가능
- 경쟁 시스템 대비 정보 풍부

**단점**:
- 경고 배너 필수 (화면 복잡도 증가)
- 오해 가능성 (확정값으로 착각)

**구현 영향**:
- Premium notice 강제 표시 로직 유지
- 법적 검토 필요

#### 옵션 2: 제공하되 더 강한 경고

**장점**:
- 법적 리스크 최소화
- 고객 오해 방지

**단점**:
- 경고가 너무 강하면 정보 가치 저하
- UX 저해

**구현 영향**:
- 경고 UI 강화 (빨간색, 팝업 등)
- 개발 영향 미미

#### 옵션 3: 제공하지 않음 (기능 제거)

**장점**:
- 법적 리스크 제거
- 화면 간결

**단점**:
- 고객 정보 부족 (보험료는 주요 관심사)
- 경쟁력 저하

**구현 영향**:
- 예제 1 제거
- 개발 영향 없음 (오히려 단순화)

### 디폴트 추천안

✅ **옵션 1 (제공 - 참고용만)**

**이유**:
- 고객 니즈 충족 (보험료는 핵심 정보)
- 현재 경고 UI가 충분히 명확
- 법적 검토 후 문구 조정 가능

**단, 조건**:
- 법무팀 검토 필수
- 경고 문구 최종 승인 필요

## D3: 원문 보기 (PDF) 구현 우선순위

### 왜 결정이 필요한가?

Evidence 모달에서 "원문 보기" 버튼이 있지만 현재는 placeholder입니다.
실제 PDF 뷰어 연결은 개발 리소스가 필요하므로 우선순위 결정이 필요합니다.

### 선택지

#### 옵션 1: 1차 구현에 필수 포함

**장점**:
- 완전한 정보 제공
- 고객 신뢰도 향상

**단점**:
- PDF 뷰어 라이브러리 선택 + 테스트 필요
- 개발 2~3주 추가
- PDF 파일 관리 인프라 필요

**구현 영향**:
- PDF 뷰어 (예: PDF.js) 연동
- PDF 파일 스토리지 (S3 등)
- 페이지 하이라이팅 기능 (선택)

#### 옵션 2: 2차 구현으로 미룸 (추천) ⭐

**장점**:
- 1차 MVP 빠른 출시
- 핵심 기능 (비교) 먼저 검증

**단점**:
- 초기 고객 경험 제한적

**구현 영향**:
- 1차: Evidence 출처 텍스트만 표시
- 2차: PDF 뷰어 추가

#### 옵션 3: 제공하지 않음 (Evidence 출처만 표시)

**장점**:
- 개발 복잡도 최소
- 유지보수 부담 없음

**단점**:
- 고객이 직접 PDF 찾아야 함
- 정보 접근성 저하

**구현 영향**:
- "원문 보기" 버튼 제거
- Evidence 출처만 텍스트로 표시

### 디폴트 추천안

✅ **옵션 2 (2차 구현으로 미룸)**

**이유**:
- 1차 MVP는 핵심 비교 기능 검증이 목표
- PDF 뷰어는 부가 기능 (필수 아님)
- 고객 피드백 받은 후 우선순위 재조정 가능

**1차 대안**:
- Evidence 출처에 "(약관 p.12)" 표시
- 외부 링크로 PDF 다운로드 제공 (간단)

## D4: Notes 상세 기본 상태

### 왜 결정이 필요한가?

현재 Notes 상세 7개 항목이 기본적으로 "접힌 상태"인데,
정보 노출 정도에 따라 고객 경험이 달라집니다.

### 선택지

#### 옵션 1: 접힌 상태 (현재 방식) ⭐ 추천

**장점**:
- 첫 화면 간결
- 필요한 사람만 펼침

**단점**:
- 중요 정보 놓칠 가능성

**구현 영향**:
- 변경 없음

#### 옵션 2: 펼쳐진 상태 (항상 표시)

**장점**:
- 모든 정보 즉시 노출
- 클릭 불필요

**단점**:
- 화면 길어짐
- 스크롤 증가

**구현 영향**:
- 기본 상태만 변경 (1줄 수정)

#### 옵션 3: 요약만 표시 (상세 제거)

**장점**:
- 가장 간결

**단점**:
- 정보 부족

**구현 영향**:
- Details 섹션 제거

### 디폴트 추천안

✅ **옵션 1 (접힌 상태)**

**이유**:
- ChatGPT 스타일과 일치
- 정보 밀도 조절 가능
- 고객이 원하면 쉽게 펼칠 수 있음

## D5: 회사명 + 상품명 표시 위치

### 왜 결정이 필요한가?

현재 "📋 비교 대상 상품" 섹션이 응답 상단에 있는데,
테이블 바로 위로 이동하거나 화면 최상단으로 고정하는 방안도 있습니다.

### 선택지

#### 옵션 1: 현재 위치 유지 (응답 상단) ⭐ 추천

**장점**:
- 논리적 흐름 (질문 → 대상 상품 → 비교)
- 모든 응답에 일관성

**단점**:
- 테이블 볼 때 스크롤 필요

**구현 영향**:
- 변경 없음

#### 옵션 2: 테이블 바로 위

**장점**:
- 테이블과 회사명 근접 배치

**단점**:
- 논리적 흐름 깨짐

**구현 영향**:
- 렌더링 순서 변경 (미미)

#### 옵션 3: 화면 최상단 고정 (sticky)

**장점**:
- 항상 보임

**단점**:
- 화면 영역 차지

**구현 영향**:
- CSS sticky 추가

### 디폴트 추천안

✅ **옵션 1 (현재 위치 유지)**

**이유**:
- 프로토타입 피드백에서 문제 없음
- 일관된 정보 구조 유지

## D6: 모바일 우선 개발 여부

### 왜 결정이 필요한가?

데스크톱 먼저 개발할지, 모바일 먼저 개발할지에 따라
프레임워크 선택 및 개발 일정이 달라집니다.

### 선택지

#### 옵션 1: 데스크톱 먼저 ⭐ 추천

**장점**:
- 복잡한 테이블 표시 용이
- 개발 속도 빠름

**단점**:
- 모바일 사용자 대응 지연

**구현 영향**:
- React + Desktop-first CSS
- 모바일은 2차

#### 옵션 2: 모바일 먼저

**장점**:
- 모바일 사용자 조기 확보

**단점**:
- 테이블 레이아웃 제약
- 개발 복잡도 증가

**구현 영향**:
- React Native 또는 Mobile-first CSS
- 개발 기간 1.5배

#### 옵션 3: 반응형 (동시 지원)

**장점**:
- 모든 기기 대응

**단점**:
- 개발 복잡도 최대
- 테스트 범위 증가

**구현 영향**:
- Tailwind CSS 등 반응형 프레임워크
- 개발 기간 2배

### 디폴트 추천안

✅ **옵션 1 (데스크톱 먼저)**

**이유**:
- 보험 비교는 주로 데스크톱 환경 (복잡한 의사결정)
- 빠른 MVP 검증 필요
- 모바일은 고객 니즈 확인 후 추가

**단, 조건**:
- 모바일 최소 지원 (테이블 가로 스크롤)
- 2차 개선에서 모바일 UX 최적화
